# Changelog (Unreleased)

Record image-affecting changes to `manager/`, `worker/`, `openclaw-base/` here before the next release.

---

- feat(manager): add Team Leader heartbeat and worker lifecycle builtins for team-scoped sleep/wake coordination (unreleased)

- fix(manager): make find-skills use deterministic script paths in worker/copaw SKILL.md, render canonical install/search commands from `hiclaw-find-skill.sh`, and treat "import/install xxx skill from market" as a direct install trigger

- chore(base): pin higress/all-in-one base image tag from `sha-d32debd` to `2.2.1` in `openclaw-base/Dockerfile`, `hiclaw-controller/Dockerfile.embedded`, and `manager/docker-legacy/Dockerfile.copaw-all-in-one`

- fix(manager): start Higress WASM plugin-server (`/usr/share/nginx/html/plugins/...` on port 8002) from the same nginx instance as Element Web. Our supervisord configs (both embedded and legacy) override the base image's supervisord which previously launched a separate `plugin-server` program — without it Envoy cannot fetch `ai-proxy`/`key-auth`/`ai-statistics` WASM modules and forwards LLM requests to upstream without Host header rewrite, resulting in 404s from the LLM backend. Listens on both IPv4 and IPv6 because Envoy resolves `localhost` to `::1` first.

- fix(worker): stop Worker-side sync from pushing `SOUL.md`, `AGENTS.md`, and `HEARTBEAT.md` back to MinIO. These files are authoritatively written by the controller (via inline `spec.soul`/`spec.agents` or package contents); the conditional `mc cp`-if-newer loop in `worker-entrypoint.sh` raced with controller pushes — for example `hiclaw apply -f` of a Worker with inline overrides would be reverted by a stale Worker push shortly after, leaving MinIO with the ORIGINAL package content. The corresponding `mc mirror` exclude list already covered these files; the individual re-push was added speculatively for "personality evolution" but no code path actually writes these files from the Worker side.

- chore(manager): drop the `hiclaw-controller` istiod debug push workaround (`GET 127.0.0.1:15014/debug/adsz?push=true`) and its `gateway.Config.PilotURL` plumbing. The original Higress all-in-one image had a bug where Console writes were not promptly pushed to Envoy in embedded mode, so the controller manually kicked istiod after consumer/route updates; this is fixed in the pinned `higress/all-in-one:2.2.1` base, and the manual trigger is no longer needed.

- fix(manager): always exclude `SOUL.md`, `AGENTS.md`, and `HEARTBEAT.md` from the initial local→OSS mirror in `Deployer.DeployWorkerConfig`. Each of these files has a dedicated authoritative writer (PutObject for SOUL.md, `prepareAndPushAgentsMD` for AGENTS.md, `pushBuiltinTopLevelFiles` for HEARTBEAT.md). Previously the mirror only excluded SOUL.md/AGENTS.md when the spec carried inline overrides; on subsequent reconciles (e.g. finalizer add or status bump) the mirror re-pushed the stale local AGENTS.md (wrapped with empty builtin markers by `executor.DeployToMinIO`) over the correctly-merged version that `prepareAndPushAgentsMD` had just written, transiently exposing missing `## Every Session` content and a missing team-context block. This is the root cause of `test-17-worker-config-verify` flakes where assertions on AGENTS.md content failed right after worker creation but passed after a subsequent update operation.

- fix(worker): restore the per-file `mc cp`-if-newer push-back of `SOUL.md`, `AGENTS.md`, and `HEARTBEAT.md` in `worker-entrypoint.sh` that commit 394a7d3 deleted. The deletion was based on the claim that "no code path actually writes these files from the Worker side", but the agent itself is the writer for the documented self-edit use cases — HEARTBEAT.md checklist tweaks (`worker-agent/AGENTS.md`: "you are free to edit HEARTBEAT.md") and SOUL.md "personality evolution". Without push-back those edits were silently lost on container restart or any subsequent file-sync pull. The original `-nt ${PULL_MARKER}` guard is sufficient on its own: every pull path (startup mirror, retry mirror, file-sync skill) touches `${PULL_MARKER}` strictly after `mc mirror` returns, so freshly-pulled package content always has mtime ≤ marker and only post-pull agent edits trigger the push. Also fix the 5-minute Remote→Local fallback loop which forgot to `touch ${PULL_MARKER}` after pulling `openclaw.json`/`skills/`, causing the change-triggered push loop's `find -newer` trigger to fire on every iteration.
- refactor(manager): remove `workers-registry.json` maintenance from worker-management skill; worker status is now queried via `hiclaw get workers`

- fix(controller): make `hiclaw apply worker --zip` honor the worker runtime. Previously `applyWorkerZip` only sent `name/model/package` in the create/update body, so `req.Runtime` was always empty and `defaultRuntime("")` defaulted everything to `openclaw`. As a side effect the integration-test matrix expansion `(controller-cr, SHARD_C_TESTS, copaw)` was running a hidden openclaw worker for `test-15-import-worker-zip`, hiding bugs that only surface in the real copaw runtime. The CLI now (a) reads `worker.runtime` (and bare `runtime`) from the ZIP's `manifest.json`, (b) honors an explicit `--runtime` flag override on top of that, and (c) forwards the value via `runtime` in the API body. The matching CI matrix passes `HICLAW_DEFAULT_WORKER_RUNTIME=${{ matrix.runtime }}` so the test ZIP's manifest declares the correct runtime per shard, and `test-15` now asserts the resulting Worker CR's runtime matches the manifest to catch any future regression.
- chore(openclaw-base): bump openclaw dependency from `hiclaw-v1.1` to `hiclaw-2026.4.14` (commit `b181285a`); pulls in the matrix SSRF fixes that allow private/loopback homeservers like `http://hiclaw-controller:6167` without an explicit `dangerouslyAllowPrivateNetwork` opt-in, and adds an explicit `curl` retry fallback for `matrix-sdk-crypto-nodejs` native binaries

- chore(openclaw-base): bump openclaw to `2f35b6fa` (atop `hiclaw-2026.4.14`) to pick up the matrix channel-restart race fix that surfaced as `Matrix sync entered STOPPED during startup` whenever the controller updated `channels.matrix.groupAllowFrom` (e.g. when a new worker was added to the team) at the same moment the manager's `message` tool fired an outbound send. The openclaw fix has three coordinated parts: (1) `client-bootstrap.ts` routes `readiness:"started"` callers through `acquireSharedMatrixClient({ startClient: true })` so two concurrent acquirers no longer race on `client.start()`; (2) the matrix channel declares `allowFrom` / `dm.allowFrom` / `groupAllowFrom` / `groupAllowFromFallbackToAllowFrom` as `noopPrefixes` so writes to those paths no longer trigger a full channel restart (the matrix monitor handler already re-reads them per message from live config); (3) `config-reload-plan.ts` emits per-plugin `noopPrefixes` before `configPrefixes` so the more-specific dynamic-read paths can override the broader `channels.matrix` hot rule under first-match-wins. Together these eliminate `test-06-multi-worker` failures where Bob was provisioned successfully but the manager could not send him the welcome message.

- chore(openclaw-base): bump openclaw to `ece35b6a` (atop `hiclaw-2026.4.14`) to pick up structured matrix-plugin trace logging gated by `OPENCLAW_MATRIX_DEBUG=1`. Hiclaw exposes the toggle via `HICLAW_MATRIX_DEBUG=1`: the install script forwards it to the controller / manager containers, the controller `WorkerEnvBuilder` propagates it to every spawned worker + manager, and the worker / manager entrypoints translate it to `OPENCLAW_MATRIX_DEBUG=1` before starting `openclaw gateway run`. Used to diagnose "worker did not join room" / "manager replied empty" hangs by emitting INFO-level JSON traces (sync state transitions, room invite/join, message handler arrival + filter outcomes, dispatch begin/settled) into the openclaw log file without a separate `--verbose` flag or a global `logging.level=debug`.

- chore(openclaw-base): rebase from `higress/all-in-one:2.2.1` (~1.79 GB) onto `higress/ubuntu:24.04` (~103 MB) — net ~1.7 GB shrink for `openclaw-base` and every downstream image (`hiclaw-manager`, `hiclaw-worker`, `cloud-manager`). The all-in-one base was originally chosen for layer sharing with the legacy single-container manager, but the modern split architecture runs Higress only inside `hiclaw-controller/Dockerfile.embedded` (which keeps its independent `FROM all-in-one:2.2.1`) and never inside the manager / worker — `start-manager-agent.sh` skips local Higress setup whenever `HICLAW_RUNTIME != docker`, and the embedded controller starts the manager with `HICLAW_RUNTIME=k8s` (via `WorkerEnvBuilder.BuildManager`). APT mirror sed now handles both legacy `/etc/apt/sources.list` and the deb822 `/etc/apt/sources.list.d/ubuntu.sources` that Ubuntu 24.04 ships. Also drops `nginx` from the base apt list — the only consumers (`setup-higress.sh`, `start-element-web.sh`) only ever run inside the embedded controller image (which installs its own nginx) or the deleted legacy all-in-one image, never in the modern manager / worker. Verified that Node.js 22 + `@matrix-org/matrix-sdk-crypto-nodejs@0.4.0` prebuilt native addon loads cleanly on glibc 2.39, and that `test-02-create-worker` + `test-15-import-worker-zip` both pass end-to-end on the new base ([f63bf98](https://github.com/agentscope-ai/HiClaw/commit/f63bf98063847570fe4e0fb608392253d68fde5d)).

- chore(legacy): delete `manager/docker-legacy/Dockerfile.all-in-one`. It bundled supervisord with `start-apiserver.sh` / `start-pilot.sh` / `start-gateway.sh` / `start-console.sh` from the all-in-one base and would no longer build after the openclaw-base rebase to plain Ubuntu 24.04. The file was not built by CI (`build.yml` only pushes `manager`, `manager-copaw`, `worker`, `copaw-worker`, `hiclaw-controller`) and the `install/hiclaw-install.sh` legacy fallback path only triggers on missing `hiclaw-embedded` images — in which case it pulls the previously-released baked image, never rebuilding from this Dockerfile. Sibling files (`Dockerfile.aliyun`, `Dockerfile.copaw-all-in-one`) remain: the former does not run any local infrastructure (cloud-only, `HICLAW_RUNTIME=aliyun`) and works on the new base unchanged; the latter was already independently `FROM all-in-one:2.2.1` and does not depend on `openclaw-base` ([f63bf98](https://github.com/agentscope-ai/HiClaw/commit/f63bf98063847570fe4e0fb608392253d68fde5d)).

- fix(manager): write `channels.matrix.userId = "@manager:${HICLAW_MATRIX_DOMAIN}"` in both `manager-openclaw.json.tmpl` and the in-place `start-manager-agent.sh` jq update path, so OpenClaw's matrix-auto-reply does not fall back to the `@default` localpart and silently drop incoming DM messages from the admin user

- fix(controller): re-pin `WorkerName: "manager"` in `DeployManagerConfig` so the generated Manager `openclaw.json` always uses `userId=@manager:<domain>` regardless of the Manager CR name (typically `default`). The controller refactor in `d029ed2` accidentally reverted commit `3f8f84b`, which made the controller-pushed `openclaw.json` overwrite the manager-rendered one with `userId=@default:<domain>` shortly after startup. OpenClaw's Matrix client then filtered every admin DM to the wrong localpart, so the agent silently received zero messages and `test-02-create-worker` failed with `LLM Calls: 0` even though the gateway, sync and access token were healthy.

- fix(controller): write `channels.matrix.network.dangerouslyAllowPrivateNetwork = true` into every controller-generated `openclaw.json` (workers and manager). The `openclaw` 2026.4.x bump rolled the matrix SSRF fix forward but the new auto-detection only treats `localhost`, raw IP literals, single-label hostnames, and `.local`/`.internal` suffixes as private — the embedded homeserver is reached via `matrix-local.hiclaw.io:18080` (a public-looking FQDN that resolves to `127.0.0.1`) and k8s workloads via cluster service DNS, so neither path opts into private-network access by default. Without this flag every freshly-provisioned worker fails to `/sync` against the homeserver, never reaches `PREPARED`, never joins its room and never replies — surfaced in CI as `test-15-import-worker-zip` "Worker did not join room within 120s" and as `test-02/03/04/06` "Manager replied (value is empty or null)" once the same regression hit the manager-side controller-rewritten config.
- fix(controller+manager): restore OpenClaw Control UI access at host port 18888 after the openclaw 2026.4.x upgrade.
  - **Port unification**: openclaw 2026.4.x merged the Control UI HTTP server into the gateway port (older versions ran a separate Control UI listener on 18799 alongside the gateway WS on 18800). The controller's hardcoded `gateway.port: 18800` left the new unified listener on 18800 while every downstream consumer — Manager Dockerfile `EXPOSE 18799`, the host:`HICLAW_PORT_MANAGER_CONSOLE`→container:18799 port mapping in `manager_reconcile_container.go`, the legacy nginx reverse proxy in `start-element-web.sh`, and the `tests/lib/test-helpers.sh` / `install/hiclaw-verify.sh` health probes — kept addressing 18799 and got connection-refused. Pin `gateway.port: 18799` in `agentconfig/generator.go`, `manager-openclaw.json.tmpl`, the in-place `start-manager-agent.sh` jq merge, and the worker template skill reference.
  - **Bind exposure**: openclaw 2026.4.x also flipped the default bind address from `0.0.0.0` to `127.0.0.1`. In hiclaw's embedded dual-container mode the manager runs in its own container and the host port mapping (host:18888 → manager-container:18799) cannot reach a loopback-only listener inside the container — connections were accepted by the container runtime's port forwarder and immediately reset because nothing was actually listening on the container's interface. Set `gateway.bind = "lan"` so the gateway binds to `0.0.0.0`; access remains gated by the shared `gateway.auth.token`.
  - **Browser auth gates**: with non-loopback binds, openclaw 2026.4.x also requires explicit `gateway.controlUi.allowInsecureAuth = true` (HTTP-only console) and `gateway.controlUi.allowedOrigins = ["*"]` (hiclaw cannot enumerate every legitimate origin since the user picks `HICLAW_PORT_MANAGER_CONSOLE` and may reach the console via 127.0.0.1, LAN IP, or custom hostname). Add both alongside the existing `dangerouslyDisableDeviceAuth = true` flag, and re-emit the full controlUi block in the controller-pushed config so `mc mirror` no longer strips it.

- fix(manager+controller): pin `channels.matrix.autoJoin = "always"` in the manager template, the controller-generated worker/manager `openclaw.json`, and the in-place `start-manager-agent.sh` jq merges. openclaw 2026.4.x defaults the matrix invite-handler policy to `"off"`, meaning agents see incoming invites in `membership:invite` and never accept them. Hiclaw's provisioning flow always invites the agent (manager or worker) into its dedicated room and depends on the agent's matrix client to auto-join — without this flag the manager / worker sits in `invite` forever, the matrix server delivers no room events to the agent (no /sync state for an unjoined room), and every dependent test (`test-02` onwards "Manager replied empty", `test-15` "Worker did not join room") fails. Verified by sending a DM after a fresh `install-embedded`: manager joins the room within one sync cycle and replies normally. Per-room access remains gated by the matrix server's invite ACL plus our existing `dm.policy=allowlist` / `groupPolicy=allowlist` allowlists, so `autoJoin=always` does not widen the trust boundary.

- fix(controller): stop rotating Matrix access tokens and gateway shared secrets on every reconcile, which used to trigger an `openclaw` gateway restart every ~5 minutes (controller's reconcile period) and tear down in-flight worker dispatches.
  - `agentconfig/generator.go` now sets `gateway.auth.token` and `gateway.remote.token` to the stable `req.GatewayKey` (already persisted alongside the Matrix password in `WorkerCredentials`), instead of generating a fresh hex string per call. The unused `generateRandomHex` helper and its `crypto/rand` / `encoding/hex` imports are removed. Aligns the controller-rendered `openclaw.json` with what the manager template (`manager-openclaw.json.tmpl`) already does, so the controller can no longer "fight" the manager-side renderer over the value.
  - `service/credentials.go` adds a persisted `MatrixToken` field to `WorkerCredentials` (saved into the on-disk env file and the in-cluster Secret).
  - `service/provisioner.go` caches the access token issued by the initial `EnsureUser` into `creds.MatrixToken` for both `ProvisionWorker` and `ProvisionManager`, and `RefreshCredentials` / `RefreshManagerCredentials` now reuse the cached token via a new `ensureMatrixToken` helper, only falling back to `matrix.Login` (and re-persisting) when the cache is empty. Without this, every reconcile issued a brand-new device session and pushed the new `channels.matrix.accessToken` into the manager's `openclaw.json` via the shared filesystem mount, which `openclaw` interpreted as a config change requiring a matrix-client (and in practice a full gateway) restart — visible in `matrix-trace` logs as the client cycling through `STOPPED` → recreated every 5 minutes and dropping mid-flight messages.

- fix(manager+controller): make YOLO mode actually trigger auto-confirm in multi-worker projects.
  - `start-manager-agent.sh` now promotes the workspace `yolo-mode` marker file into the `HICLAW_YOLO=1` env var at boot, so the agent's documented env-var check works even when the file is touched before the container starts.
  - `WorkerEnvBuilder` (controller) propagates `HICLAW_YOLO=1` from the controller's own env to every Manager and Worker container it provisions; previously the env var was lost across the controller→manager boundary in embedded mode.
  - `manager/agent/AGENTS.md` (and copaw variant) replaces the one-line YOLO note with an explicit per-session check command and documents that `~/yolo-mode` resolves to `/root/manager-workspace/yolo-mode` (since `HOME=/root/manager-workspace`).
  - `project-management/SKILL.md` adds a YOLO gotcha at the top of the list, and `references/create-project.md` hoists the YOLO check to a new Step 0 with a hard rule that Step 1c MUST auto-confirm and never post a "please confirm" question. Previously the YOLO note was a `>` blockquote between Step 1c and 1d that the LLM routinely missed, causing `test-06-multi-worker` to stall in `planning` status forever waiting for the unreachable test admin to confirm.
